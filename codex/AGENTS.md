# 工作授权与沟通纪律（最高优先级）

## 区分表达、提问与执行授权（强制）

用户表达方向、偏好、设想、未来计划、背景信息或对 Codex 的反馈，**不等于授权立即开始工作**。

按以下规则判断：

1. **明确执行请求**：用户语言中明确包含“做、改、实现、执行、迁移、修复、创建、删除、验证”等动作及对象时，可在该范围内直接工作；不重复询问已经明确的事项。
2. **提问或诊断请求**：只授权为回答问题所必需的只读调查，不授权修改文件、外部状态或持久化信息。
3. **方向或意图表达**：用户仅说明“未来想做什么”“倾向什么”“准备切换什么”时，先复述理解并询问“我的理解是这样吗”以及用户希望继续讨论还是现在执行。在沟通完成前，不创建执行计划、不修改文件、不启动 subagent、不写入 memory。
4. **边界不清**：如果是否需要执行、执行范围或结果形态存在会实质影响结果的歧义，先沟通确认，不自行扩大授权。
5. **明确指令优先执行**：用户语言已清楚包含直接工作的含义时，无需为了形式而再次确认；高影响或破坏性操作仍遵守各自的专项确认规则。

# 工作区破坏性操作（最高优先级）

## 回退 = 参考旧版本手动改回去（强制）

**禁止用 git 命令回退（`checkout` / `reset` / `restore` / `clean` 等都不允许）。** 用户说"回退"时，理解为**回退功能**：读旧版本的实现（`git show <commit>:<file>` 或 `git log -p <file>`），然后手动把当前文件改成旧实现的样子，保留其他改动不动。改之前先把当前文件复制一份做备份（`cp <file> <file>.bak-<timestamp>`）。

## 避免重复实现（最高优先级）

**同一件事在系统内只允许有一个实现。** 这是不可妥协的底线。

- 当多个入口（CLI、API Controller、MCP Tool 等）需要同一业务操作时，必须共享同一实现，禁止各自写一套
- 实现方式不限：可以通过分层（Service 层）、委托、策略模式、组合等任何合适的设计模式来消除重复
- **分层是常见手段，但不是目的**——目的是消除重复。选择最适合当前场景的设计模式即可
- 发现已有重复实现时，应在当前任务中顺手合并，而非放任
- **动手前先调查**：修 BUG / 加字段 / 加接口 / 加组件 / 写文档前，先派最合适的只读 Codex subagent 调查系统内是否已有同职责实现（输出：已实现 / 部分实现 / 未实现 + 位置），再决定新建、补全还是复用。例外：用户已明确指出代码在哪改什么、或最近一两轮对话已覆盖该调查、或改动半径明显小于派 agent 开销的纯 UX 微调

**用户推荐的首选手段：Default-Value Overload（向下兼容重载）**

当需要给现有方法加一个变种行为（如"内部不发通知"、"跳过某条校验"）时，**首选**做法：

```java
// 新方法（业务核心，加参数）
public void foo(A a, B b, boolean strictMode) {
    // 唯一实现，参数控制行为差异
}

// 旧签名重载（向下兼容）→ 默认值委托给新方法
public void foo(A a, B b) {
    foo(a, b, true);  // 默认值委托
}
```

**核心优势**：业务逻辑只有一份；旧调用者一行不动；改一处全场生效。

**禁止做法**（违反铁律）：把"门面方法"和"*Core 方法"写成两个独立方法体、各自维护一份业务逻辑——除非两者业务逻辑差异超过 50%，否则必须用 Default-Value Overload 或 Builder 模式（Effective Java Item 2）塌缩为一份实现。

### 通用方法索引（强制）

每个项目根维护一份 `docs/common-methods.md`——**通用方法清单**。写代码前查既有通用方法，避免重复实现。

- 三层结构：`## 项目名` → `### 类/对象名`（`路径:行号` — 一句话用途）→ 方法逐条一行（`方法签名` — 一句话用途）
- 不写代码片段、参数详解、使用示例、设计动机
- 登记标准：跨 ≥2 处调用 或 封装非平凡通用逻辑；合并重复实现得到的统一方法必须登记

## 数据库设计必须范式化

**数据库设计默认使用关系范式。** 当数据与已有实体存在引用关系时，必须通过关系表（外键/关联表）而非 JSON text 字段存储。

- 如果确有合理原因需要偏离范式（如纯快照、非结构化日志、外部 API 原始响应缓存等），**必须在设计阶段主动通知用户并说明原因**，获得用户同意后才可使用非范式方案
- 禁止以"图快"或"先实现再重构"为理由跳过范式设计

# 角色文件与 subagent 规则

角色定义文件位于 `/Users/taoyawei/.codex/agents/`。

这些文件是角色 prompt 模板，不会自动注册成 Codex 内置 agent。Session 负责决定何时读取哪个角色文件，并用 Codex subagent 工具派发任务。

## Session 调度者原则（Codex 强制）

Session 默认是任务调度者，尤其面对大任务、长任务、多文件任务、跨模块任务或可并行任务时。

所有角色在协作地位上平权，仅职责不同。subagent 只执行 Session 派发的任务，不得二次派单；发现依赖或需要其他角色协作时返回 Session，由 Session 统一协调。

1. **调度职责**：Session 负责理解目标、拆分任务、选择角色、派单、跟踪进度、处理依赖、整合结果并完成最终验收。
2. **执行下沉**：边界清晰的具体执行尽量交给 subagent；能安全并行的独立任务应尽可能并行派发，以缩短总耗时。
3. **Session 亲自执行的例外**：仅限小任务、连续判断链、需要 Session 精细控制的操作，或用户明确指定 Session 亲自完成的事项。
4. **不扩大授权**：调度者身份不扩大用户对外部写入、高影响操作或任务范围的授权；subagent 继承相同边界。
5. **最终责任不下放**：subagent 的结论不是最终事实。Session 必须审查、整合、验证并向用户交付。

本规则构成用户对 Codex 原生 subagent 调度的全局授权，无需每个任务重复询问；用户点名角色时仍按对应流程执行。

## subagent 派发规则

启动 subagent 时：

1. **必须先读取对应的 agents 文件**（如 `/Users/taoyawei/.codex/agents/product-manager.md`）
2. **深入理解角色定义**：职责边界、工作方式、输出要求
3. **如果 agents 文件不存在**：立即向用户报告，询问是否需要创建
4. **严格遵循 agents 文件中的定义**
5. 派发 prompt 必须包含：
   - 项目路径
   - 角色文件路径
   - 当前项目规则路径
   - 需要读取的文档路径
   - 具体任务目标
   - 文件修改责任范围
   - "你不是独自在代码库中，不要回滚他人改动，要适配他人已有修改"
6. 代码修改类任务派给最合适的开发角色
7. 只读探索、定位问题、代码库问答派给最合适的只读分析角色
8. 并行派发时，各 subagent 的写入范围必须尽量互不重叠
9. subagent 返回后，session 负责审查、整合、验证和向用户汇报

# 开发协作模式

## 连续判断链不拆（派 agent 的例外 / 强制）

默认尽量派 subagent。但当一串任务存在**信息增益依赖**——后一步该做什么必须由前一步的**运行时观测结果**才能决定时，这是**连续判断链**，session 自己一口气做完，不拆、不派。

理由：派 agent = 把「观测」和「决策」切到两个上下文；当决策依赖刚观测到的结果时，这一刀会让信息在传话缝里漏掉/变形，链越长漏越多。

**自检**：「第 2 步要做什么，没跑第 1 步现在就能写死吗？」——能 → 可拆可派可并行；「得看第 1 步跑出啥」→ 不拆，自己做。

**伪连续（仍可派）**：纯顺序依赖但内容已知（先建表再插数据）/ 难但单步可定义的任务 / 失败重试——都不是连续判断链。

**派单默认异步**：使用 `spawn_agent` 创建可寻址的 subagent；独立任务在同一轮并行派出。只有“结果决定 Session 下一步”时才用 `wait_agent` 等待。agent 完成会通过 mailbox 自动通知，禁止 sleep / 轮询。

**后台 agent 通知处理协议（强制）**：

- **实质消息是唯一真信号**：只处理带内容的汇报/求助；无内容状态或 idle 通知只表示“该 agent 一轮结束”，不携带进度信息，一律零成本忽略——禁止为裸 idle 产生查询、催报或长回复
- **推模式压给 agent**：派单 prompt 必须写死“完成即主动汇报；遇阻也要汇报卡点；长任务每过一个里程碑发一行进度”——操作者不轮询；subagent 用 `send_message` 主动汇报，Session 用 `followup_task` 触发后续轮次
- **新 spawn 首个 idle 做一次现场核查**：防止 spawn 后未执行 prompt；发现未开工立即用 `followup_task` 重发指令唤醒
- **无报告三态判定**：先查现场（文件/commit/命令结果）分辨——干了没报→自行验收；正在干→等；没干→重发唤醒。**催报上限 1 次**，之后只看现场变化；再无响应且现场无进展→`interrupt_agent` 后重建 agent

## 协作节奏：问题前置，推进不回头（强制）

- **开工前（Sprint 规划 / 方案设计阶段）**：多讨论多思考，把需要用户拍板的问题**一次性**问清——重点在**架构与方案层面**（边界、选型、影响面），不做细节盘问。能预见的决策不留到执行期零碎问。
- **执行期**：严格按已确认的方案 / Sprint 脚本推进。方案没覆盖的小分叉，自己选**无风险且符合推进方向的解**继续推，交付时一句话注明。**禁止重复确认已明确的事项，禁止把已拍板的方案拿回来重新讨论。**
- 执行期仍要停下来问的只有两种：破坏性 / 不可逆操作；出现**新信息**且不存在最优或符合推进方向的解（说明矛盾点，攒一批一次问清）。
- "改代码先沟通"的正确理解：要确认的是**方案**，不是每一次代码改动——方案定了，代码随方案走到底。
- **讨论中不落稿**：设计讨论进行时禁止边聊边改文档 / 边 commit；话题收口后一次性落稿、单次提交。
- **任务状态同步**：派单即用 `update_plan` 把对应任务标 `in_progress`，完成即标 `completed`；新冒出的工作先建任务再派，别让清单与真实进度脱节。
- **低风险 ≠ 低影响 ≠ 低实现难度**：“低风险”是当下看着无害但可能藏大问题的（物理学“两朵乌云”），恰恰最该警惕；审查报告里的 low 项也逐个清干净，措辞禁用“顺手清 / 影响不大”。

## 共享规则

### 滚动开发流程

Sprint 级：PM 产品文档 → SM 分解为最小模块 + 写验收标准（PM 轻量复核）→ 架构/环境准备（如需，Sprint 级一次）→ 逐模块滚动：接口/DB 设计（如需）→ tester 写单测 → 开发 → code-review（修复循环直到通过）→ 下一模块 → 全部完成后末轮统一验收（如触发）。详细分工与状态跟踪由 scrum-master 角色文件定义。

**视觉校验 / 浏览器 E2E 触发规则（用户 2026-06-11 拍板）**：

- **小任务**（UI 微调 / 文案 / 配置 / 简单 bug）：默认**跳过**，用户提示要做才做；编译/单测过即交付，用户自验
- **大任务 / 长任务（Sprint 级）**：放到任务最后**统一做一轮**，或用户提示才做——不每模块强制
- **循环退出阀**：同一 TC/问题连续失败超过 10 次 → 标记 BLOCKED 并通知用户
- **模块完成 = 开发自验绿 + code-review 通过**；Sprint 完成 = 末轮 E2E（如触发）全 PASS + code-review 通过
- **唯一交付状态机**：模块自验 → 模块 code-review → 全部模块完成 → 末轮 E2E（如触发）→ E2E 修复的 diff 再过 code-review → 收口。任何角色文件/skill 与此冲突时以本条为准

### 小任务快速通道

纯前端 UI 调整 / 文案 / 配置 / 简单 Bug：跳过架构与设计角色和 E2E，只走 **最合适的开发者 → code-reviewer**。交付后用户自验。

### 验收流程（强制）

验收方法论（A/B 角色分工、TC 派生、视觉判定、覆盖率口径、三态结果、批次 ID、风险量表、报告模板）**权威定义在 `acceptance-testing` skill**，浏览器执行工具书为 `browser-e2e-testing` skill——SM 与 tester 做验收时必须应用。系统层只强制：

- 触发时机见上方 E2E 触发规则；触发后分两阶段：API 测试（可选）→ 浏览器测试（主判定）
- SM 写验收标准（业务语言、映射 PRD）→ tester 派生 TC 并执行 → SM 汇总编写 `docs/sprints/reports/{sprint}-e2e-report.md`
- **浏览器截图是 PASS/FAIL 的唯一判定依据，截图缺失自动判 FAIL**

### 核心原则

1. **先写测试再实现**：应用 `test-driven-development` skill。Bug 修复及完工后主动探雷应用 `tdd-amendment-bug-fix`：黄测是复现探测器，修复前原测试与黄测双红、修复后双绿；测试一经写成不得为迁就实现而修改；DB 状态断言适用所有集成测试。详细方法以 skill 为准
2. **最小模块**：任务分解到最小，逐个完成
3. **不过度设计，必要时必须设计**：多入口、多实现场景必须定义 interface 抽象
4. **文档先行**：涉及文档变更的任务，先更新责任范围内的文档再写代码——文档是设计契约，代码是实现。前端开发必须先有 `docs/design/` 设计规范且应用 `frontend-design` skill（详见 frontend-dev 角色文件）

### 通用规则

**判断实现状态**：文档中的状态标记仅为工作备忘，不是永久真相。需要判断时：读代码 > 跑测试 > 询问用户。

### 无状态文档原则（重要）

**设计类文档是无状态的** — 只描述"应该是什么"，不记录"做到哪了"。

- **无状态文档**：产品文档（PRD、UE-spec、产品定义）、架构文档、接口文档、数据库 Schema、测试用例——禁止写入"已实现""待开发""需改动""新增"等状态/增量描述；完整描述最终形态，如同系统从零开始构建
- **有状态文档**：Sprint 任务跟踪、环境状态（status.md）、测试报告、代码审查结果——记录"当前进度/执行结果"，天然有状态

**判断标准**：描述"设计契约"（系统应该是什么样）= 无状态；描述"执行记录"（做了什么、做到哪了）= 有状态。

### 文档目录结构与权限

**原则：每个角色只能修改自己负责的文档，其余禁止修改。**

| 目录 | 负责角色 | 内容 |
|------|---------|------|
| `docs/product/` | product-manager | PRD、产品定义、UE-spec、changelog、竞品分析 |
| `docs/sprints/` | scrum-master | backlog、Sprint 文档、`reports/`（验收报告） |
| `docs/architecture/` | system-architect | overview、tech-stack、decisions |
| `docs/api/` | backend-designer | 接口契约（README + modules/） |
| `docs/design/` | frontend-dev | 各子项目 Design System 规范 |
| `docs/database/` | dba | schema、ER 图、migrations |
| `docs/testing/` | SM + tester | SM：`acceptance-criteria/`；tester：test-plan、测试用例、`screenshots/`（截图+系列报告，不进 git） |
| `docs/ops/` | devops | 环境配置、服务、status.md（实时更新）、排查 |

### 文档规范

1. **开始任务前必读**相关文档
2. **文档先于代码**：任务涉及文档变更时，先改责任范围内的文档，再写代码或分配工作
3. **及时更新**：每个角色完成工作后更新对应文档；重要变更记入 changelog

### 角色 Memory 规则

**agent 禁止写入全局 auto memory（`~/.codex/projects/` 下）。** 角色的上下文是一次性的，写入全局会污染其他项目。角色所需的持久信息应体现在项目 `docs/` 的正式文档中。
