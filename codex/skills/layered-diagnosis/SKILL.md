# 分层诊断 —— 自底向上定位故障

测试一个功能、排查一个 BUG，本质都是回答同一个问题：**"期望发生的事"到底在哪一环没发生？** 这个 skill 给出回答它的唯一可靠方法——**自底向上分层核验，不跳级**。

## 为什么必须分层（核心动机，先内化再用）

一个有数据流的功能，从底到顶要穿过好几层：数据落库 → 后端查询/返回 → 前端拿到并渲染 → 页面视觉呈现。任何一层断了，**最终现象都长一个样**（页面空、结果错、按钮没反应）。

只看最终现象（黑盒）去猜原因，会**误判 BUG 位置、误解 BUG 成因**，然后为了"让现象正确"盲目改代码——改在错的层、理解也错——结果**代码越改越错**：改坏了本来对的逻辑，或用 hack 盖住真问题，真故障还在原地。

分层的价值是：**先把故障真实钉死在某一层，再只改那一层。** 它把"我以为的原因"换成"证据指向的原因"。这是防止"为呈现而瞎改、把系统弄得更乱"的根本手段，不是流程洁癖。

> **一句话**：不分层就动手改，是在你猜错的层上动刀。分层先定位，是在 BUG 真正所在的层上动刀。

## 四层核验顺序

对每个要验的"期望"，按**从底到顶**逐层核验。每层都要**留下可复核的证据**，**前一层符合期望，才往上走一层**：

| # | 层 | 验什么 | 证据形式 |
|---|----|--------|---------|
| 1 | **数据层（DB / 存储）** | 数据是否真按期望落库 / 变更？ | `psql` / 存储查询的真实输出 |
| 2 | **接口层（API）** | 后端对这个请求返回什么？符合期望吗？ | `curl` 打接口的完整响应（含 HTTP 状态码 + body） |
| 3 | **渲染层（前端拿到的数据）** | 前端实际收到的数据 / DOM 文本对吗？ | 网络面板里前端收到的响应、`page.inner_text()`——不是 HTML 源码硬断 |
| 4 | **视觉层（呈现）** | 页面 / 客户端最终呈现对吗？ | 截图（视觉判定的最终依据） |

非 Web 的故障同理映射：第 3 层是"上层消费方拿到的数据对不对"（另一个服务、CLI 输出、消息队列消费者……），第 4 层是"最终对外行为对不对"。层的**名字**可变，**自底向上、逐级、留证据、不跳级**的骨架不变。

### 已知某层 PASS 就别重验它

如果某层已经被证据确认 PASS（比如你刚 `psql` 查过数据在），**直接从它上面一层开始**，不要回头重验。分层不是每次从头跑一遍，是"沿着证据链往上推进到第一个 FAIL 的层"。

### 一次只往上推进一层

定位阶段，**一刀只切一层**：验完这一层、拿到这一层的判定（PASS/FAIL），再决定要不要往上。不要一上来就把数据、接口、前端全查一遍——那等于退回黑盒撒网。一层一刀，就知道故障在这层还是更上层。

## FAIL 时：报告必须指出断在哪一层

定位的产出不是"它坏了"，而是**"它断在第 N 层"**。报告要明确：

> 例：「数据层 PASS、接口层 PASS、但渲染层前端拿到的字段为空 → 故障在前端渲染/序列化，**不用去翻后端**。」

这样修复方直奔故障层，不用全栈乱读代码。一份合格的分层诊断报告，读完就知道**该派谁、改哪一层、不用碰哪些层**。

## 真实案例（这套方法论怎么救场）

一次真实排查："admin 入站列表页明明 DB 有 138 条数据，页面却显示'暂无帖子'。"

**不分层时的瞎猜（全错）**——凭直觉依次猜了三个原因，每个都像模像样，每个都错：
1. "大概是 react-query 缓存没失效" → 让用户硬刷新 → 还是空。错。
2. "可能 api 连错库了" → 准备去查 datasource。
3. "估计是前端 BAODIAN_API_BASE 指向了别的实例" → 准备撒网查前端配置。

**分层后的一刀定位**：
- 数据层：已 PASS（`psql` 查到 138 条在本地库）。→ 不重验，从接口层开始。
- 接口层：带 token `curl` 打前端实际发的那个请求 `GET /api/admin/social/inbound/posts?page=1&pageSize=20`。**一刀切下去**：返回 **HTTP 500** `{"success":false,"message":"服务器内部错误"}`。

接口层 FAIL，定位结束。**根因和三个直觉全部无关**：不是缓存、不是连错库（datasource 一查正是那 138 条所在的库）、不是前端配置。是后端 SQL 在 keyword=null 时生成 `lower(?)`，PostgreSQL 把无类型占位符推断成 `bytea`，报 `function lower(bytea) does not exist` 导致整个查询 500。

**教训**：黑盒猜了三轮全错，还差点为错误的原因去改前端配置（在错的层动刀）。分层一刀就钉死在接口层，报告直接告诉修复方"故障在后端查询，前端别碰"。**省下的不是时间，是避免把对的代码改坏。**

## 与"截图是验收唯一依据"不冲突

在做客户端验收时，"截图是 PASS/FAIL 的最终视觉判定依据 + 缺图自动 FAIL"依然成立——那是**视觉层（第 4 层）**的判定规则。前三层（数据 / 接口 / 渲染）是**定位手段**，让 FAIL 时知道去哪修。四层缺一不可：没有前三层，截图 FAIL 了你只知道"不对"，不知道"哪不对"。

## 边界 —— 不要把分层变成顽疾

分层逐级**只对穿透多层、有数据流的功能**走。判断标准：

- ✅ **要分层**：功能跨了 DB→后端→前端（或等价的多环），现象在顶层但成因可能在任何一层。Sprint 级长任务、复现"结果不对但不知道为什么"的 BUG。
- ❌ **不强求分层，验最关键那一层即可**：纯 UI 微调 / 改文案 / 改样式 / 纯配置 / 一次性脚本 / 单点改动（一个常量、一处校验、一个查询换源）——没有跨层数据流，一层就够，直接验最终效果。

机械地对一个改文案的任务套"数据→接口→渲染→视觉"是浪费。这套方法论是为"现象和成因隔着好几层"的场景准备的解毒剂，不是每个任务的仪式。

## 与其他 skill 的关系

- **`browser-e2e-testing`**：Web 浏览器执行工具书（Playwright 脚本、截图函数 `shot()`、DB 验证 `db()`、选择器避坑、报告模板）。本 skill 提供"分层定位"的方法论，那个 skill 提供 Web 场景下"怎么留每一层证据"的具体工具。做 Web E2E 时两者配合用。
- **`test-driven-development` / `tdd-amendment-bug-fix`**：TDD 的"红→绿逐级推进"和本 skill 是同一原理的两面——TDD 在**写**的时候逐级（每完成一级当场自验再往上），本 skill 在**查**的时候逐级（每验一层定位再往上）。两者都为了"故障在产生/暴露的那一层当场被抓，不累积到最后才暴露"。
