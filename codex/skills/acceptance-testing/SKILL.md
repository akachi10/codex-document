# 验收测试方法论 — 基于视觉的客户端验收

本 skill 是验收工作的**完整工作流**：从验收标准到最终验收报告的所有步骤、文档、判定规则。

## 何时使用本 skill

**触发条件**（任一满足即应用）：

- 一个模块/功能开发完成，需要做 Sprint 级别的验收（本 skill 是全局交付状态机中的"末轮 E2E"环节，见"与交付状态机的关系"）
- 用户说"验收"、"E2E 验收"、"做验收测试"、"按验收标准跑测试"
- SM 派 tester 做客户端验收（不是日常单元/集成/API 测试）
- 修复 BUG 后的回归验收
- Sprint 结束前的整体验收

**不应使用的场景**（不要用本 skill）：

- 日常单元测试 / 集成测试（tester 的日常白盒工作，按本职流程跑，不需要 skill）
- API 测试 + DB 对比（tester 的独立工作，按白盒规则跑）
- 临时调试一下浏览器问题（直接用 `browser-e2e-testing` skill 跑 Playwright，不需要走完整验收流程）
- 性能 / 安全 / 可访问性专项测试（按对应专项方法论跑，本 skill 不覆盖）

## 谁应用本 skill（角色映射）

本 skill 定义的两个角色（A 验收者、B 测试者）需要由具体角色承担。在 ReLoop 项目中：

| 本 skill 中的角色 | 由谁扮演 | 在哪触发 |
|------------------|---------|---------|
| **A（验收者）** | **scrum-master**（SM agent） | SM 在 Sprint 流程的"协调验收"步骤应用本 skill |
| **B（测试者）** | **tester**（tester agent） | SM 派 tester 做客户端验收时，tester 应用本 skill |

未来其他项目用本 skill 时可指定不同的扮演者，但 A/B 的职责定义不变。

## 如何使用本 skill（5 个步骤）

### 1. SM（A 角色）启动验收

模块开发完成后，SM 主动应用本 skill，开始"第 1 步：A 写验收标准"。

### 2. SM 写完 AC 后派 tester

按"第 2-3 步"选择测试技术、拆系列、生成派单 prompt。每个系列派一个 tester subagent（同一条消息并行）。

### 3. tester（B 角色）执行客户端验收

tester 收到派单 prompt 后，按"第 4–6 步"派生 TC、执行、做视觉判定。**执行环节按客户端类型选择对应的执行 skill**（第 6 步）：

- **Web 浏览器** → 应用 `browser-e2e-testing` skill（Playwright 工具方法、截图函数、选择器规则、报告模板）
- **Flutter App**（未来） → 应用对应 Flutter 验收 skill
- **其他客户端** → 按用户提供的工具适配

### 4. tester 出测试报告

按"第 7 步"产出 `s{N}-acceptance-report.md`，含 TC + 截图 + DB 验证 + 覆盖率自检。

### 5. SM 汇总验收报告

按"第 8 步"做覆盖率审计、风险打权复核审计（优先级已在第 2 步派单前算好，此步只复核），输出 `docs/sprints/reports/{sprint}-e2e-report.md`。本 skill 的验收是全局交付状态机中的"末轮 E2E"环节，其在整条流水线中的位置见下方"与交付状态机的关系"。

## 与 `browser-e2e-testing` skill 的关系

本 skill 是**验收方法论**（流程 + 文档 + 判定规则）；`browser-e2e-testing` 是 **Web 浏览器执行工具书**（Playwright 怎么用）。

- 本 skill 在"第 6 步：客户端执行"明确指引：Web 客户端→必须应用 `browser-e2e-testing` skill 的工具方法
- 两个 skill 是**组合关系**——`acceptance-testing` 负责"为什么测、测什么、怎么判定"，`browser-e2e-testing` 负责"怎么操作浏览器"
- 两者**不重复内容**——遇到工具问题查 `browser-e2e-testing`，遇到流程问题查 `acceptance-testing`

## 设计意图（先读这条）

整个工作流的**单一主线**是：**基于视觉证据的可追溯验收**。

- "可追溯"——每条产品需求可向下追溯到验收标准、测试用例、截图、DB 字段；任何 FAIL/MISSING 可向上溯源到具体需求
- "基于视觉证据"——客户端验收的 PASS/FAIL 由真实渲染截图判定，不靠口头汇报、不靠 DOM/源码推断
- ISTQB / ISO 29119 / Risk-Based Testing 都是服务于这条主线的工具，不是目的

读这份 skill 遇到歧义时，**回到这条主线判断**。

## 两个角色（A 和 B）

本 skill 的所有协作发生在两个角色之间：

| 代号 | 名称 | 职责 |
|------|------|------|
| **A** | **验收者** | 写验收标准、派单给 B、检查 B 的报告、做最终验收判定 |
| **B** | **测试者** | 基于 A 的验收标准派生测试用例、在客户端执行、产出带截图的测试报告 |

> A 和 B 的实际人/agent 由调用方决定（在 ReLoop 项目中：A=SM、B=tester）。本 skill 不绑定具体身份。

**核心约束**：
- A 写"应该达成什么"，B 决定"怎么验证达成"
- A 不写测试用例，B 不修改验收标准
- 一切交流通过文档进行（异步、可追溯）

## 完整工作流

```
A 出验收标准（业务语言）
   ↓ 文档：docs/testing/acceptance-criteria/{sprint}.md
   ↓
A 选测试方法论 + 算执行优先级(P0/P1/P2) + 拆系列 + 生成批次 ID
   ↓ 执行优先级在拆系列/派单前算好，随派单带给 B
   ↓
A 派单给 B（每个系列一个 B 实例，并行）
   ↓ 派单 prompt 包含：验收标准 + 单一测试技术 + 系列范围 + 应用场景预估清单 + 业务影响等级 + 执行优先级
   ↓
B 派生 TC + 在客户端执行 + 截图 + DB 验证 + 出测试报告
   ↓ 文档：docs/testing/screenshots/{批次ID}/s{N}-acceptance-report.md
   ↓ 测试用例（TC）内嵌在报告中，不单独成档
   ↓
A 基于测试报告做最终验收
   ↓ 验收依据：截图（视觉）+ 三态结果 + 覆盖率审计
   ↓ 文档：docs/sprints/reports/{sprint}-e2e-report.md
   ↓
末轮 E2E 全 PASS → E2E 修复的 diff 再过 code-review → 收口；
有 FAIL/MISSING → 修复或补实现 → B 重测（全局交付状态机见"与交付状态机的关系"）
```

## 与交付状态机的关系

本 skill 的验收**不是**"E2E 通过后才进入 code-review"——那是错误口径。以全局 `~/.codex/AGENTS.md`「唯一交付状态机」为权威：

> 模块自验绿 → 模块 code-review → 全部模块完成 → 末轮 E2E（如触发） → E2E 修复的 diff 再过 code-review → 收口

- **code-review 在每个模块自验后就发生**，早于末轮 E2E；模块 code-review 通过是模块完成的一部分。
- **本 skill 描述的验收 = 上面的"末轮 E2E"环节**（全部模块完成后触发，触发规则见全局 `AGENTS.md`「E2E 触发规则」）。
- 末轮 E2E 若产生修复 diff，该 diff **再过一轮 code-review** 才收口。
- 因此本 skill 中任何"验收 PASS 决定是否放行 code-review"的旧表述，一律以本节的全局状态机为准——验收 PASS 是**收口**前的一环，不是 code-review 的前置门。

## 术语速查表

| 术语 | 含义 | 谁产出 |
|------|------|--------|
| **AC（验收标准）** | 业务语言描述"做完什么样算 OK"，每条 AC 必须追溯到产品需求编号 | A |
| **TC（测试用例）** | 基于 AC 的可执行测试步骤，**内嵌在测试报告里**，不单独成档 | B |
| **系列（Series）** | 按业务功能域聚合的 AC 集合（S1~SN），由一个 B 负责 | A 拆分 |
| **批次（Batch）** | 一次完整测试执行的标识，格式 `YYYYMMDD-HHMM` | A 生成 |
| **子批次** | 修复后重测的标识，格式 `s{N}-r{轮次}`（如 `s1-r2`） | A 生成 |
| **三态结果** | TC 的执行结果：PASS / FAIL（实现错） / MISSING（开发漏实现）。**禁止 SKIP** | B 判定 |
| **覆盖率分母** | A 派单时给出的"应用场景预估清单"总数，决定 ≥95% 的分母 | A 给出 |

## 第 1 步：A 写验收标准

A 读产品需求文档（PRD / UE-spec / 现有代码），用**业务语言**描述每个功能点"做完什么样算 OK"，输出到 `docs/testing/acceptance-criteria/{sprint}.md`。

### 验收标准每条包含的字段

| 字段 | 含义 | 示例 |
|------|------|------|
| 验收标准 ID | 唯一标识 | `F-USE-AS-PURCHASE-PRICE` |
| 源 PRD 编号 | 强制双向映射，可反查 PRD 具体需求 | `PRD-F-007` |
| 功能点标题 | 一句话概括 | "报价单作为采购价格来源" |
| 业务标准 | 业务语言描述"做完什么样算 OK"。**禁止写测试步骤** | "用户在编辑页能勾选'采购时作为价格来源'，保存后默认报价单的扫码入库会用报价单匹配价填充 inventory.price 和 reference_price" |
| 操作路径 | 用户从 UI 怎么走到这个结果 | "报价单列表 → 编辑页 → 勾选 Checkbox → 保存 → 切到默认报价单 → 扫码入库" |
| 验收点 | 视觉/数据观察点，每个点必须对应至少一张截图 | (1) 编辑页 Checkbox 显示已勾选；(2) 列表页"作为采购价"列显示 ✓；(3) 入库后 `inventory.price` 等于报价单匹配价 |
| 业务影响等级 | critical / high / medium / low（按"业务影响等级客观规则表"，见下）——定测试深度 | high |
| 涉及字段 | 用于执行优先级计算和覆盖率审计 | `quotation.use_as_purchase_price`, `inventory.price` |
| 适用测试技术 | A 在第 2 步决定，记录于此 | "等价类划分 + 状态迁移" |
| 执行优先级 | A 在第 2 步（拆系列/派单前）按"执行优先级客观规则表"算出并回填，记录于此 | P0 |
| 反例（不应发生的行为） | 来自 PRD 的负向需求，触发错误猜测技术派生 TC | "未勾选时不应填充 reference_price" |

### 两个独立维度：业务影响 × 执行优先级（必须分清）

本 skill 用**两张相互独立的规则表**（业务影响表 + 执行优先级表），二者**不做一一映射**：

| 维度 | 取值 | 定什么 | 何时计算 |
|------|------|--------|----------|
| **业务影响维度** | critical / high / medium / low | 测试**深度**——单条 AC 派生多少 TC、边界穷举到什么程度、重测几轮 | 第 1 步写 AC 时 |
| **执行优先级维度** | P0 / P1 / P2 | 执行**顺序与裁剪线**——先跑哪些、时间/资源紧张时后跑哪些 | **第 2 步拆系列、派单之前**（见下） |

**关键声明**：业务影响与执行优先级是**两个正交维度**，不存在 "critical→P0、low→P2" 这类固定对应。一条 low 业务影响的 AC 可能因其它原因排到较高优先级，反之亦然。两者各按自己的客观规则表独立打分。

**底线（跨两维统一）**：无论业务影响多低、优先级多靠后，**每条 AC（含 low 业务影响 / P2）至少有 1 条核心 TC 被实际执行**——优先级只裁剪"深度与轮次"，绝不把某条 AC 的 TC 数裁到 0。这与"全部 AC 必须被覆盖、禁止 SKIP"自洽。

#### 组合矩阵（拆系列 / 派单时二者如何联用）

| 业务影响 → / 优先级 ↓ | critical / high | medium | low |
|----------------------|-----------------|--------|-----|
| **P0** | 全量 TC + 边界穷举 + 多轮回归 | 核心 TC + 主要边界 | ≥1 核心 TC（不做边界穷举） |
| **P1** | 全量 TC + 边界穷举 | 核心 TC + 主要边界 | ≥1 核心 TC |
| **P2** | 核心 TC + 主要边界 | 核心 TC | ≥1 核心 TC（只跑单条核心路径） |

- **业务影响定"测多深"**（列方向）：critical/high 穷举边界 + 多轮；low 只跑核心路径、不做边界穷举。
- **优先级定"先跑谁、紧张时裁哪些的深度"**（行方向）：P0 先跑、深度拉满；P2 后跑、深度收敛——但**任何格子的下限都是 ≥1 条核心 TC**，不存在"整条 AC 不测"的格子。

### 业务影响等级客观规则表（消除主观判断）

按以下规则**第一条命中即终止**：

| 规则 | 业务影响等级 |
|------|---------|
| 触及金额字段 / 支付 / 退款 / 计费 | **critical** |
| 触及权限/角色控制 / 跨组织访问 | **critical** |
| 触及核心数据写入（创建/删除业务对象） | **high** |
| 触及核心数据更新（修改业务对象关键字段） | **high** |
| 触及流程状态迁移 | **medium** |
| 纯展示 / i18n / UI 文案 / 视觉样式 | **low** |
| 其他默认 | **medium** |

### 执行优先级客观规则表（第 2 步派单前计算，纯规则驱动）

**优先级必须在拆系列、派单之前算好**——它决定各系列的执行顺序与裁剪线，因此要在派单 prompt 里就带给 B，不能等到汇总阶段才算。按以下规则**第一条命中即终止**：

| 规则 | 执行优先级 |
|------|---------|
| 触及金额字段 | **P0** |
| 触及权限/角色控制 | **P0** |
| 触及核心数据写入 | **P1** |
| 触及 i18n / UI 文案 / 视觉样式 | **P2** |
| 其他默认 | **P1** |

- 字段触及判定以 AC 的"涉及字段"列为依据。
- 以上是简化版 RBT，专为字段命中规则定制，便于机械执行。完整 RBT（ISO/IEC/IEEE 29119 Part 1）需要"概率 × 影响"二维评估——本工作流暂不引入。
- 第 8 步的"风险打权审计"是对本次计算结果的**复核审计**，不是首次计算（见第 8 步）。

### 验收标准 vs 测试用例（必须分清）

| 概念 | 谁产出 | 文档 | 内容 | 语言 |
|------|--------|------|------|------|
| **验收标准** | A | `docs/testing/acceptance-criteria/{sprint}.md` | 业务侧"做完什么样算 OK" | 业务语言 |
| **测试用例（TC）** | B | **不单独成档**，直接出现在测试报告里 | 具体执行步骤、输入数据、预期视觉结果、截图、DB 验证 SQL | 执行语言 |

## 第 2 步：A 选方法论 + 算执行优先级 + 拆系列

**本步顺序**：先按"执行优先级客观规则表"（见第 1 步）给每条 AC 算好 P0/P1/P2 → 再选测试技术 → 再拆系列。**优先级必须在拆系列、派单之前算好**，因为它决定各系列的执行顺序与深度裁剪线，要随派单 prompt 带给 B。第 8 步只对本次计算做复核，不重新首算。

A 根据每个验收标准的特征，从 ISTQB 黑盒测试技术中**选择实际适用**的方法。

### ISTQB 黑盒测试技术选择器

| 功能特征 | 应用技术 |
|---------|---------|
| 离散输入域（下拉、单选、状态枚举） | **等价类划分（Equivalence Partitioning）** |
| 数值/长度/日期边界 | **边界值分析（Boundary Value Analysis）** |
| 多条件组合判定（折扣/权限/校验） | **判定表测试（Decision Table Testing）** |
| 对象有状态生命周期 | **状态迁移测试（State Transition Testing）** |
| 跨多步用户旅程 | **场景法 / 用例测试（Use Case Testing）** |
| ≥3 个独立参数组合（避免组合爆炸） | **配对测试（Pairwise / Combinatorial Testing）** |
| 任何复杂度 | **错误猜测（Error Guessing）**——异常输入、并发、网络中断、超长字符串、空值 |
| 多角色权限交叉 | 用 **判定表** 建模 role × action 矩阵 |
| i18n / 视觉规范 | 用 **等价类**（语言列表分组）+ **错误猜测**（缺失 key、超长翻译） |

**关键原则**：用到几个技术派几个 B 实例，**不固定数量**。简单功能 1-2 个，中等 3-4 个，复杂 5-6 个。

### 系列拆分

按业务功能域将验收标准拆为系列（S1~SN），每个系列指定**一个 B + 一种测试技术**。

- **原则**：一条验收标准只由一个系列覆盖（避免重叠覆盖时的冲突判定）
- **例外**：复杂功能确需多技术覆盖同一验收标准时，A 在派单时显式标注；汇总时按"全 PASS 才算 PASS、任一 FAIL 即整体 FAIL"判定

### 批次 ID 规则

A **统一生成主批次 ID**（格式 `YYYYMMDD-HHMM`），通过派单 prompt 传给所有 B。子批次 ID 也由 A 生成（如 `s1-r2`），不允许 B 自行起 ID。

---

## 第 3 步：A 派单给 B（同一条消息并行）

每个 B 实例的 prompt 必须包含以下字段。**漏给任一字段** → B 必须主动索要，**禁止猜测**或继续执行。

| 项目 | 内容 |
|------|------|
| 批次 ID | A 统一生成，格式 `YYYYMMDD-HHMM` |
| 验收标准文档路径 | `docs/testing/acceptance-criteria/{sprint}.md`（B 派生 TC 的唯一依据） |
| 系列编号 + 范围 | 如"S1：库存管理，覆盖验收标准 F-001 ~ F-005" |
| 指定的测试设计技术 | 等价类划分 / 边界值分析 / 判定表 / 状态迁移 / 场景法 / 配对 / 错误猜测 之一 |
| 技术核心规则 | 如"对每个数值/长度边界 X，生成 X-1, X, X+1 三组输入" |
| **应用场景预估清单（覆盖率分母）** | A 列出本系列下该技术应产生的应用场景列表（如边界值技术下，对字段 `price` 列出 min-1/min/min+1/max-1/max/max+1 = 6 个场景）。这是 B 报告中"覆盖率"的分母 |
| 覆盖率目标 | ≥95%——分母 = 上一行的预估清单总数；分子 = B 实际派生并执行的 TC 数 |
| 测试模式 | **基于视觉的客户端验收**——截图是 PASS/FAIL 的唯一判定依据 |
| **客户端类型** | Web / Flutter / 其他——决定 B 引用哪个执行 skill |
| 截图目录 | `docs/testing/screenshots/{批次ID}/s{N}/` |
| 报告输出路径 | `docs/testing/screenshots/{批次ID}/s{N}-acceptance-report.md` |
| 环境信息 | 客户端入口 URL、后端 URL、DB 连接 |
| 数据准备方式 | 测试数据前缀约定（如 `S1-Buyer-`） |
| 测试用户准备 | 用户凭据 / 注册方式 |
| 业务影响等级 | 来自验收标准（critical/high/medium/low）——定测试深度 |
| **执行优先级** | P0/P1/P2——A 在拆系列前已按执行优先级规则表算好，随派单带给 B，定执行顺序与裁剪线 |
| **应用的执行 skill** | Web → `browser-e2e-testing`；Flutter → 对应 skill；其他 → 对应 skill |

## 第 4 步：B 派生 TC + 执行 + 出报告

### B 的工作流程

```
读验收标准 + 单一技术规则 + A 给的应用场景预估清单
  ↓
基于该技术穷举本系列所有应用场景（以 A 预估清单为分母，覆盖率 ≥95%）
  ↓
为每个场景派生 1 个 TC：步骤、输入数据、预期视觉结果、DB 验证 SQL
  ↓
启动客户端，执行每个 TC（应用对应客户端的执行 skill）
  ↓
每步截图 → 视觉对比预期 → 判定 PASS / FAIL / MISSING
  ↓
产出测试报告（含 TC 步骤 + 截图 + DB 验证 + 覆盖率自检）
  ↓
返回 A
```

### B 的禁止事项

- **禁止跨技术**：A 指定边界值，就只用边界值
- **禁止改验收标准**：有歧义找 A
- **禁止自起批次 ID**：A 统一生成
- **禁止凭文字判定**：必须有截图
- **禁止 SKIP**：每个 TC 必须有 PASS/FAIL/MISSING 之一

### B 的逃生通道（必须主动回执，不允许默默返回空报告）

| 情况 | B 行动 |
|------|--------|
| A 指定技术对该系列产生 0 个有效场景（如 Checkbox 用边界值） | 返回 **N/A 报告**（说明原因 + 给降级建议如"建议改用错误猜测"），通知 A |
| 验收标准要求但代码无对应入口/接口（开发漏实现） | 该 TC 标 **MISSING** 状态（区别于 FAIL/SKIP），单独列出告知 A |
| Prompt 字段缺失 | 主动索要，禁止继续 |

### 三态结果

| 状态 | 含义 | 后续处理 |
|------|------|----------|
| **PASS** | 视觉判定通过 + 文本断言通过 + DB 验证通过 | 标记完成 |
| **FAIL** | 实现了但行为/视觉错误 | 派开发者修 BUG，B 重测（连续失败 >10 次 → BLOCKED 通知用户） |
| **MISSING** | 验收标准要求但代码无对应入口/接口（开发漏实现） | 派开发者**补实现**（不是修 BUG），B 重测 |

## 第 5 步：基于视觉的判定（强制）

**适用范围**：本规则**仅适用于客户端验收**，不适用于 API 测试 / 单元测试 / 集成测试（后者按白盒测试规则跑，不要求截图）。

### 截图硬规则

- **每个验收 TC 必须有 `screenshot:` 字段**引用一张或多张截图，**且文件实际存在**
- **截图缺失 → 该 TC 自动判 FAIL**，不接受任何文字声明
- **每个验收点对应至少一张截图**（验收点 → 截图 一一映射）
- 截图编号命名（`001-login.png`），**内嵌到 markdown 报告**
- A 评审时**先数截图后看文字**——截图与判定文字不符直接打回

### 视觉判定 SOP（四档）

| 视觉状态 | 判定 | 说明 |
|---------|------|------|
| 元素渲染正确 + 文本可读 + 位置符合 UE-spec | **PASS** | 唯一通过路径 |
| 元素不可见（不可见状态 / 视口外 / 透明度 0） | **FAIL** | 即使存在也算失败 |
| 元素被遮挡（Modal、Toast、其他元素覆盖） | **FAIL** | 截图证据 + 标"视觉缺陷" |
| 元素渲染但错位 / 溢出 / 颜色错 / 字体溢出 | **FAIL** | 标"视觉缺陷" |
| 验收标准要求的元素**根本不存在**（代码无入口/接口） | **MISSING** | 区别于 FAIL，触发"开发漏实现"分支 |

### 三重验证（缺一不可）

- **客户端截图**（视觉，最终判据）
- **客户端可见状态断言**（按客户端类型选 API）
- **DB SQL 验证**

**禁止源码层断言**：禁止用 DOM HTML 源码或 Widget 树原始结构做最终判定——必须用**真实渲染后的可见状态**。

---

## 第 6 步：客户端执行（按客户端类型应用对应 skill）

本 skill 不重复执行细节，按客户端类型分流：

### Web 浏览器

应用 **`browser-e2e-testing` skill**——提供：
- Playwright 前置检查清单（chromium、psql、服务可用性）
- `shot()` / `db()` 函数模板（截图 + DB 查询）
- 选择器避坑表（用精确选择器、不用 has-text、用 `is_visible()` 不用 `content()`）
- Headless 默认开启
- try/finally 数据清理
- 系列报告 Markdown 模板

**Web 客户端 = 必须用 `browser-e2e-testing` skill 的工具方法**。不允许绕过或自行编写浏览器自动化流程。

### Flutter App（未来）

待引入时建立对应执行 skill（如 `flutter-e2e-testing`），覆盖：
- Flutter `integration_test` 或 `patrol` 框架的前置检查
- Flutter 可见状态断言 API（Finder / Matcher）
- 截图 + DB 验证函数
- 报告模板

视觉判定原则不变（截图是唯一判据），工具替换为 Flutter 生态。

### 其他客户端

按用户提供的工具适配。基本要求：
1. 能在真实客户端运行
2. 能截图
3. 能查 DB
4. 能产出 Markdown 报告

## 第 7 步：B 的报告（覆盖率自检表）

报告末尾必须包含覆盖率自检表。**分母由 A 派单时给出**（"应用场景预估清单"字段），B 不允许自定分母。

| 验收标准 ID | A 给的应用场景（分母） | TC 编号 | 截图 | 状态 |
|------------|----------------------|---------|------|------|
| F-001 | 场景 1：勾选 Checkbox 后保存 | TC-001 | 001-checked.png | ✅ PASS |
| F-001 | 场景 2：未勾选 Checkbox 保存 | TC-002 | 002-unchecked.png | ✅ PASS |
| F-001 | 场景 3：勾选后再取消 | TC-003 | 003-toggle.png | ❌ FAIL |
| F-002 | 场景 4：列表显示标记 | （无） | （无） | ⚠️ MISSING（代码无入口） |

**覆盖率 = (PASS + FAIL) / A 给的应用场景总数 ≥ 95%**。
- MISSING 不计入分子（开发漏实现，需补实现后重测）
- **优先级/业务影响裁剪的是"深度"（是否穷举边界、重测几轮），不是"是否覆盖该 AC"**：每条 AC（含 low 业务影响 / P2 优先级）都必须至少有 1 条核心 TC 进入分子。低优先级下**被裁掉的边界穷举场景**（如 low 域不做的 min-1/max+1 等边界）不计入分母也不计入分子；但**该 AC 的核心路径 TC 始终计入**，不允许整条 AC 从覆盖率表中缺席。
- <95% 必须说明原因（场景对该技术不适用 / 阻塞无法执行 / MISSING 比例过高），A 会判断是否打回补测

## 第 8 步：A 汇总验收报告

各 B 报告返回后，A 汇总到 `docs/sprints/reports/{sprint}-e2e-report.md`：

```
1. 收集各系列报告的 PASS/FAIL/MISSING 数
2. 失败项跟踪：列出所有 FAIL 的 TC、修复轮次、最终状态
3. MISSING 项跟踪：列出所有 MISSING 的验收点（开发漏实现），派开发者补实现并重测
4. 截图审计（先数截图后看文字）：
   - 报告中每个 TC 必须有截图引用且文件存在
   - 截图缺失或截图与判定文字不符 → 该 TC 自动判 FAIL
   - A 至少抽查 3 个 TC 截图，确认视觉内容与 PASS 判定一致
5. 覆盖率审计（强制，反向对账）：
   - 以验收标准 ID 为锚，反查所有系列报告的覆盖率自检表
   - 每条验收标准必须出现在至少一个系列的自检表中
   - 重叠覆盖判定：全 PASS 才算 PASS，任一 FAIL 即整体 FAIL
   - 覆盖率 <95% 的系列必须重派 B 补测，禁止放行
6. 风险打权复核审计（对第 2 步已算好的优先级做复核，不是首次计算）：
   - 执行优先级已在**第 2 步派单前**按"执行优先级客观规则表"算好并带给了 B（P0/P1/P2）。本步只做**复核**：抽查各 AC 的优先级判定是否与"涉及字段"列一致，发现错判则纠正并要求相应系列按新优先级补/减深度。
   - 业务影响维度控制的是**测试深度**（TC 数量与轮次），**不取消任何 AC 的最低覆盖**：
     - critical/high 业务影响：全量 TC + 边界穷举 + 多轮回归
     - medium 业务影响：核心 TC + 主要边界（P2 优先级下可不做边界穷举，仍跑核心 TC）
     - **low 业务影响：至少 1 条核心 TC 被实际执行**（只跑单条核心路径、不做边界穷举）——**绝不裁到 0 条**
   - 底线：无论业务影响或优先级如何组合，每条 AC 至少 1 条核心 TC 落地执行，与"全部 AC 必须被覆盖、禁止 SKIP"自洽。深度的伸缩见"组合矩阵"（第 1 步）。
   - 字段触及判定以验收标准的"涉及字段"列为依据。
```

### 验收报告模板

```markdown
# 验收报告：{Sprint 名称}

## 测试系列安排

| 系列 | 功能域 | 验收标准范围 | 测试技术 | B 实例 | 数据前缀 |
|------|--------|-------------|---------|--------|----------|

## 测试批次

| 轮次 | 批次 ID | 时间 | 范围 | 结果 |
|------|---------|------|------|------|

## 各系列结果

| 系列 | 测试技术 | 总数 | 通过 | 失败 | MISSING | 覆盖率 | 报告 |
|------|---------|------|------|------|---------|--------|------|

## 覆盖率审计（验收标准 × TC 追溯）

| 验收标准 ID | 源 PRD 编号 | 业务标准 | 覆盖系列 | 覆盖 TC | 综合判定 |
|------------|-------------|---------|----------|---------|----------|

## PRD 需求级覆盖率

| PRD 编号 | PRD 需求 | 对应验收标准 | 覆盖? |
|---------|---------|-------------|------|

## 验收结论
全部 PASS / 有 N 个 BLOCKED 项 / 有 N 个 MISSING 待补实现
```

### A 放行前必须检查

1. **打开测试报告**：确认每个 TC 有截图、有 DB 验证、有 PASS/FAIL/MISSING 判定
2. **抽查截图**：至少抽查 3 个 TC 的截图，确认截图内容与判定结论一致
3. **确认无 SKIP**：报告中不存在任何 SKIP 状态
4. **确认覆盖率**：每个系列覆盖率 ≥95%，所有验收标准都被覆盖

---

## FAIL 处理策略（系列级别）

修复和重测的粒度是**系列级别**，不是全局级别：

1. **系列完成后立即处理 FAIL**：某系列跑完有 FAIL → A 立即派开发者修复，不等其他系列完成
2. **修复与其他系列并行**：S1 的修复和 S2/S3 的测试是并行的
3. **非阻塞 BUG**：B 记录 FAIL 并继续执行完整个系列，系列结束后 A 派开发者集中修复
4. **阻塞 BUG**：B 中断系列并立即报告 A，A 立即派开发者修复，修复后 B 从断点继续

### 修复重测批次规则（子批次）

修复后的重测使用**子批次**，在主批次目录下创建子目录，主批次不变：

```
docs/testing/screenshots/20260415-1700/     # 主批次
├── s1/                                     # S1 初始测试
├── s1-acceptance-report.md                 # S1 初始报告（历史记录，不可改）
├── s1-r2/                                  # S1 修复后重测（子批次）
├── s1-r2-acceptance-report.md              # S1 重测报告
├── s2/                                     # S2 一次通过
└── s2-acceptance-report.md
```

- **子批次 ID**：`s{N}-r{轮次}`（如 `s1-r2` = S1 第 2 轮重测）
- **子批次只重测 FAIL + 受影响的回归 TC**，不需要重跑全部
- A 在验收报告中记录：初始批次 FAIL → 子批次修复验证 PASS，追溯链完整

---

## 退出与升级条件

- 单 TC 连续失败 >10 次 → A 标记 BLOCKED 并通知用户
- 需求歧义无法用任一技术建模 → A 停下来问用户，禁止凭空猜
- B 报告覆盖率 <95% → A 打回 B 补测，不允许放行
- 任一技术的 B 产出 0 个 TC（说明该技术对此功能不适用）→ A 在汇总报告中明确说明"不适用"，不作为缺陷

| Web 浏览器自动化（Playwright 函数、选择器避坑、headless、目录布局、报告 Markdown 形态） | `browser-e2e-testing` skill |
