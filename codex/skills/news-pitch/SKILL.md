# news-pitch — 选题立项

把"我想发一篇关于 X 的文章"从想法变成可执行的立项记录。

## 何时使用

- 用户主动提出选题（"发一篇关于 OpenAI 新模型的稿"）
- 主编自己浏览信息源后想做某个题目
- 收到外部投稿需要判断要不要做
- 需要登记被毙掉的选题（避免重复立项）

## 三问筛选

立项前先过这三关。**任一答 No 就毙掉或推迟**：

### 问 1：是热点 / 财经 / 科技 之一吗？

参考项目 persona 的范畴（编辑部承认的选题主线）。**不是范畴内的直接毙**——主线之外的题目本编辑部不做。

### 问 2：信源是否能找到至少 2 个独立来源？

- 一手权威发布（公司公告 / 监管文件 / 学术论文 / 官方账号）算 1 个独立来源
- 主流媒体的报道算 1 个独立来源
- 互相转载的信息**不**算独立来源
- 单一爆料 / 网传 / 未署名内部消息 → 直接毙

### 问 3：一句话能向普通读者讲清"这事跟你有什么关系"吗？

如果讲不清"为什么要看这篇"，说明选题没穿透到普通人——可能是过度内行话题或纯八卦，毙掉。

举例：
- "X 公司发布 Y 模型，文本生成能力提升 30%" → "AI 写作 / 编程效率会提升，可能影响入门级文案 / 程序员就业"（OK）
- "Z 协议升级到 v2.3" → 普通读者完全无法解码（毙）

## 立项登记表

通过三问后，新建 `docs/maintainer/pitches/{YYYY-MM-DD}-{slug}.md`（或项目 persona 指定的等价路径，slug 用英文小写连字符）。**不要混进产品 / 工程文档目录**。

模板：

```markdown
# Pitch: <文章标题草案>

- **状态**：pitch
- **立项时间**：<YYYY-MM-DD HH:MM>
- **类型**：快讯 / 报道 / 长读
- **预估字数**：<300 / 800-1500 / 2000-4000>
- **板块预选**：<code>（来自 GET /api/sections，附决策理由）

## Hook

<50 字内说清"这事是什么、为什么值得发">

## 三问筛选记录

1. 主线归属：热点 / 财经 / 科技 — <理由>
2. 信源清单（至少 2 个独立）：
   - 信源 A：<URL / 名称 / 性质>
   - 信源 B：<URL / 名称 / 性质>
3. 普通读者关切：<一句话 "这事跟你有什么关系">

## 信源调查计划

- [ ] 一手发布（公告 / 论文 / 官方账号）—— 链接：
- [ ] 主流媒体核对 —— 链接：
- [ ] 反方 / 不同口径来源 —— 链接：
- [ ] 数据 / 图表来源 —— 链接：

## 配图计划

- [ ] 封面（16:9，匹配主旨）：<来源 / 拍摄 / 生成方式>
- [ ] 正文图至少 1 张：<内容计划>

## 主编观点（如有）

<是否打算在文末加「主编观点」段？预计观点是什么？>

## 状态历史

- <YYYY-MM-DD HH:MM> pitch（立项）
```

## 状态机

文章一生只走这一条状态机：

```
pitch  →  drafting  →  fact-checking  →  archived  →  published
  │           │             │
  └───────────┴─────────────┴──→  killed （终态，记录原因）
```

任一非终态都可能被毙：`pitch`（信源不够 / 无对口板块）、`drafting`（写不下去 / 时效过）、`fact-checking`（核查不过 / 触红线）。`killed` 是终态。

| 状态 | 含义 |
|---|---|
| `pitch` | 立项登记，尚未动笔 |
| `drafting` | 在写（用 `news-writing` 持续修订） |
| `fact-checking` | 初稿完成，进入 `fact-checking` skill |
| `archived` | 外链已存档（`source-archiving` 完成），即将 submit |
| `published` | 已通过 approve，对外可见 |
| `killed` | 立项后被毙（信源不够 / 时效过 / 红线 / 价值低） |

每次状态变更在登记表的「状态历史」追加一行。

## 板块预选规则

**不写死板块名。** 立项时：

1. 调项目板块查询接口（如 `GET /api/sections` 或 persona / credentials 文档里指定的等价端点）拿当前列表
2. 按文章主旨匹配最贴近的 `code`，把 code 写到「板块预选」字段
3. 如有歧义，写两个候选 code，等写到 80% 完稿时再定
4. 文章主旨不属于任何板块语义 → killed，理由"无对口板块"

## killed 流程

被毙的选题也要登记，避免一个月后又冒出来重新讨论同一件事被毙：

- 同一个 pitch 文件，状态改 `killed`，文末加 `## Kill Reason` 段说明
- 不删除 pitch 文件
- 同一题材 30 天内再次被提，先搜 pitches 目录看是否已 killed

## 自检

立项完成后，回答：

1. 三问全部答 Yes 了吗？
2. 信源清单写满了吗？至少 2 个独立来源？
3. 板块预选 code 在接口返回结果里能找到吗？
4. 配图计划具体到来源了吗（不是写"配一张图"敷衍）？

任一项答否 → 回去补，不要进入 drafting 状态。

## 反例（什么时候不要用本 skill）

- 已经在写正文了 → 用 `news-writing`
- 准备 submit 了 → 用 `fact-checking` + `source-archiving`
- 写代码 / 改文档 / 跑测试 → 不要用本 skill，回退到 SM 派对应 agent
