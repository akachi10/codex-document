# fact-checking — 事实核查

文章发布前最后一道关。**没过这关不准 submit。**

## 何时使用

- 初稿写完，准备进入发布流程
- 突发新闻 30 分钟内要发
- 已发文章被读者 / 内部发现事实错误，需要写 Correction
- 复盘上一篇错误时回查

## 七步核查流程

### 1. Claim Extraction（声明提取）

把文章里**所有可验证的声明**列出来。声明 = 任何带"是 / 不是 / 多少 / 何时 / 谁说 / 怎么样"的陈述。

类型：

| 类型 | 例子 | 优先级 |
|---|---|---|
| 数字 / 数据 | "营收同比增长 23.5%" / "2026 年 4 月发布" | 最高，错了显眼 |
| 当事人 / 头衔 | "前 CEO 张三 / 现任 CFO 李四" | 高，错了贻笑大方 |
| 时间 / 地点 | "4 月 25 日在北京" | 高 |
| 因果 / 动机 | "因为 X 才导致 Y" | 中，主观判断要标 |
| 引语 | "「...」马斯克 4 月 28 日 X 帖" | 高，必须逐字核对 |
| 行业 / 历史背景 | "这是该公司近三年最大并购" | 中 |

### 2. Research（调研）

每条声明回去查证据：

- 数字 → 回到一手数据源（财报 PDF / 监管平台 / 论文 Table）
- 头衔 → 公司官网 IR 页 / LinkedIn 官方账号 / 监管文件
- 时间地点 → 主流媒体多家报道交叉
- 引语 → 原帖 URL（不要从 Twitter 截图复制，要打开 X 看原文）

### 3. Evidence（证据强度）

每条声明记录证据强度：

| 强度 | 标准 |
|---|---|
| **直接证据** | 一手原文 + URL |
| **间接证据** | 主流媒体 ≥ 2 家独立报道 + URL |
| **薄弱证据** | 单一媒体 / 未署名爆料 |
| **无证据** | 找不到来源 |

**薄弱证据**的声明必须在正文里限定（"目前仅 X 报道"）；**无证据**的声明删除。

### 4. Contact Sources（联系信源）—— 编辑部不做一手采访时通常跳过

- 国际新闻调查传统要求联系信源核实
- 如果项目 persona 声明编辑部不做一手采访（多数信息整合型门户都是），**这一步通常跳过**
- 例外：当某条引语 / 数据矛盾，可发邮件 / X DM 给当事人 / 公司 PR 求证；24 小时内无回复就在文中标"截至发稿，X 公司未回复本编辑部求证请求"

### 5. Rating（评级）

给每条声明打分（参考 PolitiFact 量表，简化为 5 级）：

| 评级 | 标准 |
|---|---|
| **True** | 证据充分，多源印证 |
| **Mostly True** | 主体正确，细节有出入 |
| **Mixed** | 一半对一半错 / 有时有效有时无效 |
| **Mostly False** | 主体错误，但有部分真实成分 |
| **False** | 完全错误 / 无证据支持 |

**只有 True / Mostly True 的声明才能发**。Mixed 必须改写并标限定；Mostly False / False 必须删除。

### 6. Documentation（文档化）

新建 `docs/maintainer/fact-checks/{article-slug}.md`（或项目 persona 指定的等价路径）：

```markdown
# Fact Check: <文章标题>

- **关联文章**：<pitches/...md slug>
- **核查时间**：<YYYY-MM-DD HH:MM>
- **核查员**：editor-in-chief agent

## 声明清单

### Claim 1
- **正文位置**：<段落起始字句>
- **声明**：<完整陈述>
- **类型**：数字 / 头衔 / 时间地点 / 因果 / 引语 / 背景
- **证据**：
  - 直接证据：<URL>
  - 间接证据：<URL>
- **强度**：直接 / 间接 / 薄弱 / 无
- **评级**：True / Mostly True / Mixed / Mostly False / False
- **处理**：通过 / 改写并限定 / 删除

### Claim 2
（同上）

## 总结

- 总声明数：N
- True / Mostly True：N
- Mixed（已限定）：N
- 删除：N
- 是否可发布：Yes / No
```

### 7. Correction Mechanism（更正机制）

发布后被发现错误，**必须**走 Correction：

1. 在原文末尾**永久**加 `## 更正` 段：

```markdown
## 更正

- <YYYY-MM-DD HH:MM>：原文 X 段提到 Y，事实应为 Z。已修正。感谢读者 / X 反馈。
```

2. **不要静默改正文**——读者已经看过的版本和现在不一样会失信
3. 如果是事实性大错（人名 / 数字差量级 / 引语张冠李戴），考虑发独立的更正声明文章并归档原文
4. 在 `docs/ops/persona-incidents.md` 留笔记（按 persona §9 要求）
5. 复盘：这次错误是核查链哪一步漏了？下次怎么避免？

## 突发新闻 30 分钟协议

突发新闻不可能完成完整七步。**简化为：**

### 0-10 分钟：抓最低事实

只发**已被 ≥ 2 家独立媒体确认**的核心事实：
- 何时
- 何地
- 谁
- 发生了什么（最简版）

正文模板：

```
<YYYY-MM-DD HH:MM> 突发：<核心事实陈述句>

据 <来源 A> 与 <来源 B> 报道，<时间地点>，<事件>。

【已确认】
- <已交叉印证的事实点 1>
- <已交叉印证的事实点 2>

【尚未确认 / 流传中但本编辑部暂无独立验证】
- <网传内容 1>
- <网传内容 2>

【后续将持续追踪】

(本文将持续更新，请关注最新版本)
```

**关键**：把"我们知道 / 我们不知道"分开写，给读者明确的不确定性边界，比硬装全知更可信。

### 10-30 分钟：补背景与多方表态

- 加历史背景（同类事件参考、当事方过往表态）
- 加多方表态（受影响方 / 监管方 / 反对方）
- 持续在文末更新「最后更新时间」

### 30 分钟后：转入正常七步

继续核查、补证据、必要时改稿（**但事实段不能静默改，要在文末「更正」段说明**）。

## Correction 决策树

发现已发文章错误：

```
错的是什么？
  ├─ 错别字 / 排版 → 静默修正 OK
  ├─ 数字 / 日期 / 名字 出错
  │     → 文末加「更正」段 + 修正正文
  ├─ 引语张冠李戴 / 篡改原意
  │     → 文末加「更正」段 + 修正正文 + persona-incidents 笔记
  ├─ 整段事实判断错误（量级错误 / 因果倒置）
  │     → 文末加「更正」段 + 修正正文 + 发独立更正声明文章 + persona-incidents
  └─ 整篇方向错（信源被骗 / 误判骗局）
        → 撤稿 + 发声明文章 + persona-incidents + 复盘
```

## 自检

七步走完后回答：

1. 所有声明评级都是 True / Mostly True 吗？
2. Mixed 的声明都加了限定吗？
3. Mostly False / False 都删了吗？
4. 有没有"我觉得是"但找不到证据的声明？（删）
5. 引语都打开了原帖 / 原文 URL 逐字比对吗？
6. 突发新闻是否清晰区分了"已确认 / 未确认"？

任一答否 → 不能 submit。

## 反例（什么时候不要用本 skill）

- 还没写完初稿（先用 `news-writing` 写完再核查）
- 信源还没体检（先 `source-verification`）
- 外链还没存档（核查通过后用 `source-archiving`）
- 主编观点段（观点不是事实，不入核查；但观点必须**基于**已核查的事实）
