# fact-checking — 事实核查

两道关：写作中引用任何素材前先做「信源核验（SIFT）」，发布前再走「七步核查」。**七步没过不准 submit。**

## 何时使用

- 初稿写完，准备进入发布流程
- 突发新闻 30 分钟内要发
- 已发文章被读者 / 内部发现事实错误，需要写 Correction
- 复盘上一篇错误时回查
- **写作中要引用某人 / 某账号 / 某图 / 某链接 / 某数据** → 先做下方「信源核验（SIFT）」

## 信源核验（SIFT）

写作中遇到要引用的内容，先做信源体检——这一步在写初稿、引用素材阶段做，先于下方的发布前七步核查。**没体检不准引用**。

**何时用**：准备引用某条 Tweet / 微博 / TG 帖子、某篇外媒报道、某张图 / 某段视频、某份数据 / 某个图表、某个个人或机构的发言。

### SIFT 四步法

每条引用都过这四步。**先 Stop，再行动。**

**S — Stop**：先别复制粘贴，问自己——
- 这条信源我熟悉吗？
- 转发链是什么？最早是谁发的？
- 我是不是因为这条内容"符合我的预期"才想引用？（确认偏误警报）

**I — Investigate the source**（调查信源本身）：

| 信源类型 | 调查项 |
|---|---|
| 个人账号 | 注册时间、历史发帖、蓝标 / 实名信息、其他平台账号是否一致、过往可信度 |
| 媒体账号 | 是否官方认证、所属机构、过往报道质量、是否有撤稿记录 |
| 机构账号 | 是否官方域名、是否有备案 / 注册信息、域名 WHOIS |
| 公司公告 | 是否官方网站 / 投资者关系页 / 监管平台（SEC EDGAR / 巨潮资讯 / HKEX）|
| 学术论文 | 是否经同行评审、期刊影响力、作者背景、是否预印本（arXiv / SSRN） |

**F — Find better coverage**（找更好的覆盖）：
- 这件事**有没有主流媒体也在报**？路透 / 彭博 / FT / 纽时 / 华尔街日报 / 澎湃 / 财新等
- 多家口径**一致**才能引；如果只有一家在报、其他没跟，先标"目前仅 X 报道"
- 如果是政府 / 公司行为，回到**一手发布渠道**（官网 / 官方公告 / 监管平台）核对原文

**T — Trace claims to original**（追溯到原始来源）：
- 二手报道 → 找一手发布
- 截图 → 找原帖 URL
- 数据 → 找原始数据集 / 报告 PDF
- 图片 → 反向图搜（Google Lens / TinEye / Yandex 图搜，见下）
- 视频 → InVID / WeVerify 拆帧反查 / 看 metadata（见下）

### 信源信誉评级

引用前给每个信源打分：

| 等级 | 标准 | 处理 |
|---|---|---|
| **高** | 一手权威（监管文件 / 官方公告 / 同行评审论文 / 当事人本人发布） | 可直接引用 |
| **中** | 主流大型媒体的署名报道，与至少 1 个独立来源交叉印证 | 可引用，需标明出处和发布时间 |
| **低** | 单一媒体报道无交叉、二手转述、未署名爆料 | 慎用，必须明示"目前仅 X 报道，本文无独立验证" |
| **不可用** | 匿名爆料 / 私人 TG 群截图 / 未实名小号 / 内容农场 | **不要引用** |

### 社交账号验真 checklist

引用 Tweet / 微博 / TG / Bluesky 帖子前过一遍：

- [ ] 账号注册时间 > 1 年（新号警报，可能是钓鱼或临时号）
- [ ] 历史发帖语言风格、发帖时区一致（不一致警报）
- [ ] 其他平台是否有同名一致的账号
- [ ] 蓝标 / 实名 / 机构认证状态
- [ ] 帖子未被作者本人删除（截图后帖子可能被删）
- [ ] 截图与原帖 URL 比对，文字、时间戳、互动数一致

### 反向图搜（Reverse Image Search）

引用图片前**必须**反向搜：

| 工具 | URL | 用途 |
|---|---|---|
| Google Lens | https://lens.google.com/ | 全球覆盖广，识别商品和地点强 |
| TinEye | https://tineye.com/ | 找到图片首次出现时间 |
| Yandex 图搜 | https://yandex.com/images/ | 在中俄内容上比 Google 更强 |

**找出图片：**
1. 第一次出现的时间和站点
2. 是否被 P 过 / 是否是合成
3. 是否被错误情境化（旧图新发）

EXIF 元数据：用 ExifTool 或在线工具读拍摄时间 / 设备 / GPS（注意社交平台通常会剥 EXIF）。

### 视频验证

| 工具 | 用途 |
|---|---|
| **InVID** | 浏览器插件，关键帧提取 + 反向图搜 |
| **WeVerify** | InVID 的衍生项目，更多元数据 |
| **YouTube DataViewer** | 通过 YouTube ID 提取视频上传时间 |

验证项：上传时间、地点是否一致（GPS / 地标 / 路牌）、画面与音频是否同步、是否有剪辑痕迹。

### 合法访问 paywall

遇到付费墙文章是常见情况。合法绕过工具：

| 工具 | 合法性 | 用法 |
|---|---|---|
| **Unpaywall** | 完全合法 | 浏览器插件，自动找该论文 / 文章的开放访问版本 |
| **图书馆数据库** | 合法 | 大学 / 公共图书馆订阅，VPN 进图书馆 IP |
| **作者预印本** | 合法 | 论文作者通常在 arXiv / SSRN / 个人主页放预印版 |
| **机构官网** | 合法 | 公司 / 政府公告通常在自家官网免费公开 |
| **Internet Archive** | 灰区 | Wayback Machine 上的快照可看，但通常不能下载 PDF |

**不合法的工具不要碰**（Sci-Hub / 12ft.io 在多国管辖下不合法 / 不稳定）。

### 信源体检表

每篇文章生成一份 `docs/maintainer/source-checks/{article-slug}.md`（或项目 persona 指定的等价路径）：

```markdown
# Source Checks: <文章标题>

- **关联文章**：<pitches/...md 的 slug>
- **核验时间**：<YYYY-MM-DD HH:MM>

## 引用清单

### 引用 1
- **正文位置**：<段落起始字句>
- **信源**：<URL / 名称>
- **信源类型**：<个人 / 媒体 / 机构 / 公告 / 论文 / 图 / 视频>
- **SIFT 走完了**：S [ ] I [ ] F [ ] T [ ]
- **信誉评级**：高 / 中 / 低 / 不可用
- **交叉印证来源**：<URL>
- **备注**：<疑点 / 待确认项>

### 引用 2
（同上）

## 总结

- 全部引用至少有几个高 / 中评级？
- 是否所有「低」评级的引用在正文里都做了"目前仅 X 报道"的限定？
- 是否有「不可用」评级被引用？（必须移除）
```

### 信源核验自检

体检完成后回答：

1. 每条引用都至少有 1 个独立来源交叉印证吗？
2. 「不可用」评级有没有被引用？（一条都不能有）
3. 「低」评级的引用都在正文做了限定声明吗？
4. 图片 / 视频都做了反向搜了吗？

任一答否 → 修引用或删段落。

## 七步核查流程

### 1. Claim Extraction（声明提取）

把文章里**所有可验证的声明**列出来。声明 = 任何带"是 / 不是 / 多少 / 何时 / 谁说 / 怎么样"的陈述。

类型：

| 类型 | 例子 | 优先级 |
|---|---|---|
| 数字 / 数据 | "营收同比增长 23.5%" / "2026 年 4 月发布" | 最高，错了显眼 |
| 当事人 / 头衔 | "前 CEO 张三 / 现任 CFO 李四" | 高，错了贻笑大方 |
| 时间 / 地点 | "4 月 25 日在北京" | 高 |
| 因果 / 动机 | "因为 X 才导致 Y" | 中，主观判断要标 |
| 引语 | "「...」马斯克 4 月 28 日 X 帖" | 高，必须逐字核对 |
| 行业 / 历史背景 | "这是该公司近三年最大并购" | 中 |

### 2. Research（调研）

每条声明回去查证据：

- 数字 → 回到一手数据源（财报 PDF / 监管平台 / 论文 Table）
- 头衔 → 公司官网 IR 页 / LinkedIn 官方账号 / 监管文件
- 时间地点 → 主流媒体多家报道交叉
- 引语 → 原帖 URL（不要从 Twitter 截图复制，要打开 X 看原文）

### 3. Evidence（证据强度）

每条声明记录证据强度：

| 强度 | 标准 |
|---|---|
| **直接证据** | 一手原文 + URL |
| **间接证据** | 主流媒体 ≥ 2 家独立报道 + URL |
| **薄弱证据** | 单一媒体 / 未署名爆料 |
| **无证据** | 找不到来源 |

**薄弱证据**的声明必须在正文里限定（"目前仅 X 报道"）；**无证据**的声明删除。

### 4. Contact Sources（联系信源）—— 编辑部不做一手采访时通常跳过

- 国际新闻调查传统要求联系信源核实
- 如果项目 persona 声明编辑部不做一手采访（多数信息整合型门户都是），**这一步通常跳过**
- 例外：当某条引语 / 数据矛盾，可发邮件 / X DM 给当事人 / 公司 PR 求证；24 小时内无回复就在文中标"截至发稿，X 公司未回复本编辑部求证请求"

### 5. Rating（评级）

给每条声明打分（参考 PolitiFact 量表，简化为 5 级）：

| 评级 | 标准 |
|---|---|
| **True** | 证据充分，多源印证 |
| **Mostly True** | 主体正确，细节有出入 |
| **Mixed** | 一半对一半错 / 有时有效有时无效 |
| **Mostly False** | 主体错误，但有部分真实成分 |
| **False** | 完全错误 / 无证据支持 |

**只有 True / Mostly True 的声明才能发**。Mixed 必须改写并标限定；Mostly False / False 必须删除。

### 6. Documentation（文档化）

新建 `docs/maintainer/fact-checks/{article-slug}.md`（或项目 persona 指定的等价路径）：

```markdown
# Fact Check: <文章标题>

- **关联文章**：<pitches/...md slug>
- **核查时间**：<YYYY-MM-DD HH:MM>
- **核查员**：editor-in-chief agent

## 声明清单

### Claim 1
- **正文位置**：<段落起始字句>
- **声明**：<完整陈述>
- **类型**：数字 / 头衔 / 时间地点 / 因果 / 引语 / 背景
- **证据**：
  - 直接证据：<URL>
  - 间接证据：<URL>
- **强度**：直接 / 间接 / 薄弱 / 无
- **评级**：True / Mostly True / Mixed / Mostly False / False
- **处理**：通过 / 改写并限定 / 删除

### Claim 2
（同上）

## 总结

- 总声明数：N
- True / Mostly True：N
- Mixed（已限定）：N
- 删除：N
- 是否可发布：Yes / No
```

### 7. Correction Mechanism（更正机制）

发布后被发现错误，**必须**走 Correction：

1. 在原文末尾**永久**加 `## 更正` 段：

```markdown
## 更正

- <YYYY-MM-DD HH:MM>：原文 X 段提到 Y，事实应为 Z。已修正。感谢读者 / X 反馈。
```

2. **不要静默改正文**——读者已经看过的版本和现在不一样会失信
3. 如果是事实性大错（人名 / 数字差量级 / 引语张冠李戴），考虑发独立的更正声明文章并归档原文
4. 在 `docs/ops/persona-incidents.md` 留笔记（按 persona §9 要求）
5. 复盘：这次错误是核查链哪一步漏了？下次怎么避免？

## 突发新闻 30 分钟协议

突发新闻不可能完成完整七步。**简化为：**

### 0-10 分钟：抓最低事实

只发**已被 ≥ 2 家独立媒体确认**的核心事实：
- 何时
- 何地
- 谁
- 发生了什么（最简版）

正文模板：

```
<YYYY-MM-DD HH:MM> 突发：<核心事实陈述句>

据 <来源 A> 与 <来源 B> 报道，<时间地点>，<事件>。

【已确认】
- <已交叉印证的事实点 1>
- <已交叉印证的事实点 2>

【尚未确认 / 流传中但本编辑部暂无独立验证】
- <网传内容 1>
- <网传内容 2>

【后续将持续追踪】

(本文将持续更新，请关注最新版本)
```

**关键**：把"我们知道 / 我们不知道"分开写，给读者明确的不确定性边界，比硬装全知更可信。

### 10-30 分钟：补背景与多方表态

- 加历史背景（同类事件参考、当事方过往表态）
- 加多方表态（受影响方 / 监管方 / 反对方）
- 持续在文末更新「最后更新时间」

### 30 分钟后：转入正常七步

继续核查、补证据、必要时改稿（**但事实段不能静默改，要在文末「更正」段说明**）。

## Correction 决策树

发现已发文章错误：

```
错的是什么？
  ├─ 错别字 / 排版 → 静默修正 OK
  ├─ 数字 / 日期 / 名字 出错
  │     → 文末加「更正」段 + 修正正文
  ├─ 引语张冠李戴 / 篡改原意
  │     → 文末加「更正」段 + 修正正文 + persona-incidents 笔记
  ├─ 整段事实判断错误（量级错误 / 因果倒置）
  │     → 文末加「更正」段 + 修正正文 + 发独立更正声明文章 + persona-incidents
  └─ 整篇方向错（信源被骗 / 误判骗局）
        → 撤稿 + 发声明文章 + persona-incidents + 复盘
```

## 七步核查自检

七步走完后回答：

1. 所有声明评级都是 True / Mostly True 吗？
2. Mixed 的声明都加了限定吗？
3. Mostly False / False 都删了吗？
4. 有没有"我觉得是"但找不到证据的声明？（删）
5. 引语都打开了原帖 / 原文 URL 逐字比对吗？
6. 突发新闻是否清晰区分了"已确认 / 未确认"？

任一答否 → 不能 submit。

## 反例（什么时候不要用本 skill）

- 还没写完初稿（先用 `news-writing` 写完再核查）
- 外链还没存档（核查通过后用 `source-archiving`）
- 主编观点段（观点不是事实，不入核查；但观点必须**基于**已核查的事实）
