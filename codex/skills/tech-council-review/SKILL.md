# 技术委员会评审（Tech Council Review）

对一份**已成型的方案**（PRD / 架构 / Sprint 计划 / 收敛方案 / 设计文档）做多视角对抗评审，在落地前把盲点、想当然的前提、技术雷挖出来。

> 这套方法论吸收自业界红队/审议实践（deliberation-debate-red-teaming 的 5 步流程 + 风险量表、adversarial-review 的对立激励去幻觉、devils-advocate 的防早期锁定），并按本方法论的取舍裁剪：**只保留技术委员会角色**（剔除财务/法务/竞品/监管/记者等商业角色），**全员平权无权重**，并新增一个业界没有的原创席位——**愚者**。

## 核心信条

1. **对立激励趋向真实**：一个 agent 被要求「找问题」会为了交差编出问题（误报）。让**提方案方**与**反对方**对立、再让**主持人**裁决，幻觉性问题会被互相抵消，结论更可信。
2. **Steelman 不 strawman**：攻击方案最强、最善意的版本，不打稻草人。经不起推敲的担忧自己先划掉。
3. **天真是一种武器**：内行把太多前提当「理所当然」咽了下去。一个不懂项目的局外人能问出内行永远问不出的问题。
4. **全员平权**：没有谁的发言天然更重。一条意见的分量由它的**风险分**决定，不由发言者的头衔决定。

## 委员会角色（固定 6 席）

| 席位 | 读不读文档 | 视角 | 由谁扮演 |
|---|---|---|---|
| **PM**（产品经理）| 读 | 用户价值、需求是否被曲解、有没有漏需求 | product-manager agent |
| **SM**（敏捷教练）| 读 | 可交付性、模块切分、依赖、范围蔓延 | scrum-master agent（或主持人自己） |
| **架构师** | 读 | 技术选型、边界、一致性、可维护性、有没有重复实现 | system-architect agent |
| **devops** | 读 | 部署、环境、上线路径、可观测、回滚 | devops agent |
| **悲观者** | 读（吃透） | 墨菲定律，技术执行层的雷（边界/时区/并发/降级/2am） | the-pessimist agent |
| **愚者** | **不读** | 塔罗 0 号牌：未被成见污染的第一性发问，戳「凭什么理所当然」 | the-fool agent |

**全员平权，无发言权重。** 收敛时按风险分排序，不按头衔。

席位可按方案性质裁剪：纯前端方案可略 devops；纯数据库方案可加 dba。但**悲观者 + 愚者两席任何评审都必须在场**——它们是对抗去幻觉的核心。

## 智能筛选其他 skill 发言（不是全部硬调）

系统里有大量 skill，多数与某次评审无关（pptx、股票 API 等）。**不要全部硬调一遍**。做法：

1. 列出当前可用 skill 清单。
2. 对每个 skill 做一句话相关性判断：「这个 skill 的专业领域，对本方案有话说吗？」
3. **有则纳入一次发言、无则弃权**。例如评审一个前端 dashboard 方案：`frontend-design`（审美）、`layered-diagnosis`（分层核验思路）相关 → 发言；`pptx`、`claude-api`、`xlsx` 无关 → 弃权。
4. 被纳入的 skill 以其专业视角对方案提**一条**最重要的意见，不展开。

筛选标准写进会议记录（哪些 skill 被纳入、哪些弃权、为什么），保证可追溯、不黑箱。

## 两档深度：基础单轮 vs 多轮深推演

按方案利害选档（沿用业界 deliberation 方法论的复杂度分级）：

| 档 | 何时用 | 用什么 |
|---|---|---|
| **基础档（单轮）** | 低-中利害、可逆、信心较高 | 下面的 5 步流程足够 |
| **多轮深度档** | 高利害 / 不可逆或回退昂贵 / 信心低 / 战略级 | 在 5 步基础上叠加 [resources/methodology.md](resources/methodology.md) 的高级技法：**pre-mortem（假设已失败、倒推原因）/ attack-tree（系统枚举失败路径）/ tabletop（注入式情景推演）/ 多轮升级提问（好奇→直接挑战→激进追问）/ Toulmin 模型验证批判真伪 / 二阶效应·反演·假设浮现**。多轮辩论可跑 2-3 轮，每轮强度递增，直到不再冒出新 showstopper 才收敛 |

**dashboard 这类高利害重写默认走多轮深度档。**

## 工作流（5 步）

复制这个清单跟踪进度：

```
技术委员会评审进度：
- [ ] Step 1: 界定方案与利害
- [ ] Step 2: 召集委员会 + 筛选相关 skill
- [ ] Step 3: 逐席发言（并行）+ 反对席红队攻击
- [ ] Step 4: 风险量表打分 + 收敛排序
- [ ] Step 5: 输出 showstopper 清单 + 修订建议
```

### Step 1 — 界定方案与利害
明确：评审对象（具体方案文档路径，不是模糊想法）、利害（做错了会怎样、可不可逆）、当前信心、是否有 deadline。利害越大，批判强度越高。

### Step 2 — 召集委员会 + 筛选 skill
- 确定 6 席中哪些到场（默认全到；按方案性质增减，但悲观者+愚者必到）。
- **席位默认由 Codex 原生 subagent 执行**：Session 把每席的视角与职责写入对应 prompt，用户在当前任务中明确点名 Claude 参与时，才可经 `claude-delegate` skill 将指定席位交由 Claude 提供异源视角；未点名时不自动调用、建议或随机混入外部引擎。无论席位由哪个授权引擎执行，发言均同等纳入风险打分（全员平权不变），会议记录注明引擎；主持人/用户保留最终决定权。
- 准备**两份不同的输入包**：
  - **给读文档的席位**（PM/SM/架构师/devops/悲观者）：方案全文 + 相关项目文档/勘探事实。
  - **给愚者**：**只有一句话需求 + 方案的结论摘要**。绝不给文档。这是铁律——愚者的价值在于不被驯化。
- 筛选相关 skill（见上节）。

### Step 3 — 逐席发言（并行）+ 红队攻击
- 各席位**并行**发言（用 `spawn_agent` 启动对应角色）。每席产出一份意见。
- 悲观者按墨菲定律推演技术雷；愚者凭直觉戳前提。
- 被纳入的 skill 各贡献一条专业意见。

### Step 4 — 风险量表打分 + 收敛
主持人（调用本 skill 的 session）汇总所有意见，**去重**，对每条用统一量表打分：

- **严重度 Severity 1-5**：5=数据损坏/线上瘫痪/钱算错 · 4=核心功能不可用 · 3=体验受损/延期 · 2=边缘 bug · 1=瑕疵
- **可能性 Likelihood 1-5**：5=>80% 触发 · 4=50-80% · 3=20-50% · 2=5-20% · 1=<5%
- **风险分 = 严重度 × 可能性**

分级（与 deliberation skill 一致）：
- **Showstopper（≥15）**：落地前必须解决
- **高优先（10-14）**：应解决或备好缓解
- **监控即可（5-9）**：接受但有预案
- **接受（<5）**：知会后放行

**对立去幻觉**：悲观者/愚者提的每条，主持人要反问「这经得起 steelman 吗？是真雷还是为反而反？」——经不起的降级或划掉，并记录划掉理由（防别人重复担心）。

### Step 5 — 输出
产出一份评审纪要（见下方模板 / resources/template.md），含：到场名单、skill 筛选记录、各席意见、风险打分表、showstopper 清单、对每条 showstopper 的修订建议、最终结论（放行/有条件放行/打回）。

## 输出模板

详见 [resources/template.md](resources/template.md)。骨架：

```
# 技术委员会评审纪要：{方案名}

## 〇 基本信息
- 评审对象 / 利害 / 信心 / deadline
- 到场席位 / 缺席及理由
- 纳入发言的 skill / 弃权的 skill（及理由）

## 一 各席意见摘要
（PM / SM / 架构师 / devops / 悲观者 / 愚者，逐席一段）

## 二 风险打分表
| # | 来源席位 | 风险描述 | S | L | 分 | 分级 |

## 三 Showstopper（≥15）逐条 + 修订建议

## 四 已划掉（steelman 后不成立，记录防重复）

## 五 结论
放行 / 有条件放行（列条件）/ 打回（列必改项）
```

## 边界（什么时候别用这个 skill）

- **不是代码审查**：评 diff/找 bug 用 `code-review`。本 skill 评的是方案文档。
- **不是验收**：验收用 `acceptance-testing`。
- **方案还没成型别用**：连草案都没有时，先去发散/设计，别空评。
- **小改动别用**：一行配置、文案微调不值得开委员会。

## 与 skill-creator 的关系

本 skill 由 skill-creator 方法论指导创建，方法论主体逐字采纳业界最成熟的开源红队 skill（lyndonkl deliberation-debate-red-teaming），并裁剪为技术委员会 + 新增愚者席。若要给本 skill 的**触发准确率**做量化调优（测试集 + 描述优化），回到 skill-creator 流程；上面的 rubric 已提供**产出质量**的量化尺子。
